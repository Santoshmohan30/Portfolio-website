Case Studies

How the work changed when reality pushed back

These deep dives are here to show problem-solving maturity: where the first design failed, what I learned, and how the system improved.

MouseBase: from a useful app idea to a maintainable backend model

Problem Statement

I needed a lab management system that could track breeding records, cage state, scheduling, procedures, and pup history without turning daily lab operations into spreadsheet chaos.

Challenges Faced

The difficulty was not building forms. It was designing a relational model that represented multiple connected entities cleanly, especially when animal state, breeding outcomes, procedures, and scheduling all overlap.

I also had to confront the limit of an early SQLite-first design. It helped fast development, but it forced me to think ahead about migrations, concurrency, and whether the schema would remain sane when the app grew.

What I Tried

Early versions leaned too heavily on simplified tables and direct feature additions. That made the UI move faster at first, but the data model became harder to reason about as relationships expanded.

Some entity boundaries were initially too coarse, which made querying and validation logic more awkward than expected.

Final Solution

I re-centered the design around normalized entities and clearer ORM boundaries, separating concerns between breeding records, procedures, scheduling, and pup lineage tracking. That improved maintainability, query logic, and future migration readiness.

Key Learnings

Good backend design is often about accepting that the first schema is a draft. I learned to design with change in mind, not just present requirements, and to think early about a path from local simplicity to PostgreSQL-grade scalability.

Why This Matters

This project reflects how I approach reliability, maintainability, and production readiness: real systems need clean models, migration thinking, and usability that does not compromise architectural clarity.

HealthBot: moving from naive retrieval to a modular agentic workflow

Problem Statement

The goal was to build a medical knowledge retrieval system that could do more than retrieve passages. It needed to structure reasoning, guard against unsafe output, and remain explainable.

Challenges Faced

Naive RAG approaches often look convincing during demos but break under edge-case questions, weak retrieval hits, or unsafe answer generation. The challenge was designing a flow that could inspect its own inputs and fail more gracefully.

What I Tried

I started with a simpler retrieval-plus-generation pattern, which made iteration easy but gave me poor control over confidence, traceability, and response safeguards. It became clear that a single-step pattern was not enough for a sensitive use case.

Final Solution

I redesigned the system into a LangGraph-based multi-stage workflow with retrieval, verification, safety validation, and response generation as explicit components. This made the architecture easier to inspect, tune, and evaluate.

Key Learnings

Strong AI systems depend on orchestration quality as much as model quality. The real work is often in boundaries, evaluation loops, and the ability to stop bad outputs before they look polished.

Why This Matters

This project shows my bias toward robust AI engineering: modular systems, explicit checks, and designs that treat reliability as a first-class requirement.

Intelligent Document Processor: engineering for messy inputs and scalable extraction

Problem Statement

I wanted a backend service that could convert PDFs and image-based documents into structured machine-readable JSON through clean APIs and a containerized processing flow.

Challenges Faced

Document inputs are inconsistent by default. OCR noise, document layout variability, and downstream schema expectations create a pipeline where each stage affects correctness and maintainability.

What I Tried

Early extraction logic was too optimistic about document consistency. That caused brittle parsing assumptions and made output normalization harder than expected.

Final Solution

I separated ingestion, OCR, normalization, storage, and API delivery into clearer stages. That made it easier to test each failure surface independently and improve extraction logic without rewriting the entire service.

Key Learnings

Backend robustness often comes from reducing hidden coupling. When services have explicit boundaries, debugging becomes faster and scaling decisions become more concrete.

Why This Matters

This project demonstrates production-oriented backend thinking: clear interfaces, operational structure, and systems that can absorb imperfect data without collapsing into manual cleanup.

Computational Neuroscience Pipeline: learning where scientific pipelines break at scale

Problem Statement

I needed an end-to-end pipeline for large-scale calcium imaging analysis that could handle ingestion, motion correction, signal extraction, normalization, deconvolution, and behavioral annotation in a reproducible way.

Challenges Faced

Large TIFF stacks, frame alignment issues, synchronization between neural and behavioral data, and memory pressure made scaling difficult. Motion correction and neural signal extraction were especially sensitive to preprocessing and data quality assumptions.

Part of the work involved recognizing that some scaling ideas were simply not working as expected and needed to be rethought rather than patched.

What I Tried

Initial pipeline passes were too monolithic and did not isolate memory-heavy stages well enough. I also learned the hard way that alignment and synchronization issues can silently degrade downstream features if validation is too shallow.

Final Solution

I moved toward a more explicit staged pipeline with stronger quality checks, clearer data flow, and targeted handling for motion correction, extraction, normalization, and behavioral frame annotation. That made both debugging and interpretation more tractable.

Key Learnings

Scientific pipelines need the same engineering discipline as production systems, but with even more respect for validation. Scaling is not only a compute problem; it is also a correctness problem.

Why This Matters

This work shows how I translate research challenges into engineering structure, with attention to reproducibility, maintainability, and scientific rigor under real data complexity.

Engineering Mindset

How I think as an engineer

System design starts with boundaries

I prefer to make responsibilities explicit early: where data enters, where validation belongs, where state lives, and which parts of the system should stay independently testable.

Modular pipelines beat opaque complexity

Whether the work is AI or scientific computing, I look for stages that can be inspected, benchmarked, replaced, and debugged without destabilizing everything else.

Debugging is structured, not reactive

I reduce uncertainty by isolating failure surfaces, verifying assumptions at interfaces, and instrumenting the pipeline before guessing at fixes.

Validation changes the quality of decisions

I care about evaluation because clean-looking outputs can still be wrong. That mindset comes from both ML workflows and scientific data pipelines where downstream trust depends on upstream rigor.

Tradeoffs should be intentional

I think carefully about performance versus simplicity, local development speed versus long-term scalability, and shipping now versus creating technical debt that will slow the next iteration.

Research ideas only matter when they survive implementation

I enjoy translating research logic into working systems. That means turning vague concepts into interfaces, data contracts, validation steps, and repeatable engineering process.
